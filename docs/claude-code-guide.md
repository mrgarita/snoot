# Claude Code 運用ガイド（プロジェクト共通指示書）

> **本ファイルを参照する Claude Code への注意**
>
> - **新プロジェクト立ち上げ時の標準フロー**（章 2 の 3 ステップを通しで実施）を採る場合は、
>   作業に入る前に本ガイドを **全章通読** すること。章 3／章 4／章 5／章 6 は互いに参照し合うため、
>   特定章だけを抜き読みすると、Why／How、`CLAUDE.md` テンプレ、権限設定の整合が崩れる。
> - **特定章だけを単独参照**することは設計上許容している（例：既存プロジェクトに memory だけ
>   追加したいので章 5 のみ読む）。その場合でも、他章にある関連事項（章 3 の Why／How、
>   章 4 の `CLAUDE.md` テンプレ、章 6 の権限例）との整合を必ず確認してから作業を進めること。

mrgarita 氏の作業環境において、Claude Code に対して恒久的に効かせたい指示と、
新規プロジェクト立ち上げ時に毎回セットアップしておきたい資産（`CLAUDE.md`、
memory ファイル、`settings.local.json`）の雛形をまとめた指示書。

monorize プロジェクトで積み上がった指示・運用を、別プロジェクトでも素早く
再現するためのリファレンスとして機能する。

## 1. はじめに

### 目的
本ガイドは、mrgarita 氏のもとで稼働する Claude Code に対して **どのプロジェクトでも
共通で適用したい運用方針** を集約したものである。新しいプロジェクトを開始する際、
本ガイドを参照すれば次の 3 点を一通り揃えられる。

- 共通の作業方針（章 3）を Claude Code に再認識させる
- リポジトリ直下に置く `CLAUDE.md` の骨格（章 4）
- Claude Code の memory 機構に保存する `feedback_*.md`（章 5）

### 想定読者
- 本ガイドのオーナー mrgarita 自身（次プロジェクトを始めるとき手順を見返す用途）
- mrgarita 氏のもとで動く Claude Code（新プロジェクトの最初のセッションで本ガイドを
  読み込み、章 5 の memory を自プロジェクト用 memory フォルダへ複製する）

### 構成の見取り図
- 章 2：新プロジェクトへの **適用 3 ステップ**
- 章 3：4 つの **共通方針サマリ**（指示の本文を一望できる）
- 章 4：`CLAUDE.md` の **プレースホルダ付きテンプレ**
- 章 5：`feedback_*.md` の **コピペ用テンプレ**
- 章 6：`.claude/settings.local.json` の **運用方針と許可項目の例**
- 章 7：本ガイド自体の **更新ルール**

## 2. 新プロジェクトへの適用手順（3 ステップ）

### 一括適用プロンプト例（推奨）

新プロジェクト立ち上げ時は、Claude Code に次のプロンプトを貼って **3 ステップを一括で
実行** させるのが標準フロー。各ステップで該当章を参照させるため、章 4／5／6 のいずれも
取りこぼさない。

> `docs/claude-code-guide.md` を全文読み、章 2 の 3 ステップを順に実行してください。
>
> - **Step 1**：章 4 を参照して、リポジトリ直下に `CLAUDE.md` を作成（プレースホルダは
>   本プロジェクトの値に置換）。
> - **Step 2**：章 5 を参照して、`~/.claude/projects/<project>/memory/` 配下に
>   `feedback_*.md` を 4 件作成し、`MEMORY.md` にインデックスを追加。
>   `{{project_name}}` と `{{existing_services}}` は本プロジェクト用に置換。
> - **Step 3**：章 6 を参照して、必要に応じて `.claude/settings.local.json` の
>   `permissions.allow` に最初の許可項目を追記。

以下の各 Step は、Claude Code が上記プロンプトを実行する際に参照する詳細。
**個別 Step のみを単独実行することも可能**（例：「memory だけ追加したい」は Step 2 のみ）。

### Step 1：リポジトリ直下に `CLAUDE.md` を作成
章 4 のテンプレをそのままコピーし、`{{...}}` のプレースホルダをプロジェクト固有
情報で埋める。章 4 の表（プレースホルダ一覧）に従えば過不足なく書ける。

### Step 2：Claude Code に共通方針 memory を保存させる
章 5 のテンプレを `~/.claude/projects/<project>/memory/` 配下に 4 件コピーし、
`MEMORY.md` にインデックスを追加する。新プロジェクト立ち上げ時は章 2 冒頭の
**一括適用プロンプト** を使うのが標準フロー。

**既存プロジェクトに memory だけ追加したい** 場合は、次のように指示する：

> `docs/claude-code-guide.md` の章 5 を読み、4 件の memory ファイルを本プロジェクト用に
> `~/.claude/projects/<project>/memory/` 配下へ保存してください。`{{project_name}}` と
> `{{existing_services}}` は本プロジェクトの値に置き換えてください。

この単独呼び出しでは Step 1（`CLAUDE.md` 作成）と Step 3（権限追記）は実施されない。
新プロジェクト立ち上げ時は **章 2 冒頭の一括適用プロンプト** の使用を推奨する。

### Step 3：必要に応じて権限を追記
`.claude/settings.local.json` の `permissions.allow` は **プロジェクトごとに
個別追加** で運用する（章 6）。最初は空のまま始め、Claude Code が許可を求めて
きたタイミングで追加していけば十分。よく使う系統は章 6 にまとめてあるので、
最初から入れておきたい場合の参考にする。

## 3. 共通方針サマリ（4 つの恒久指示）

各方針は「ルール本文」「Why」「How to apply」の 3 ブロックで構成される。
章 5 の memory テンプレと内容は同じ。ここでは一望できる形で再掲する。

### 3.1 すべての成果物を日本語で作成する

**ルール本文**：プロジェクト配下での生成テキストはすべて日本語で書く。
対象は以下を含む。

- ユーザーへのチャット返信、進捗報告、ターン末の要約
- `AskUserQuestion` の質問文・選択肢のラベルと説明
- コード内のコメント（`//`, `#`, `/* */`, docstring 等）
- コミットメッセージ、PR タイトル、PR 本文
- README、CLAUDE.md、設計メモ、その他のドキュメント類
- エラーメッセージやログ出力のうち、エンドユーザー向けに表示される文字列

例外として **英語のまま** とするのは次のみ（言語仕様・エコシステム慣習のため）。

- プログラミング言語のキーワード、標準ライブラリ／フレームワーク／パッケージの API 名
- 識別子（変数名・関数名・クラス名・ファイル名・ディレクトリ名など）
- 設定ファイルのキー名、環境変数名

**Why**：2026-05-01 に「テキストおよび受け答え」を日本語と指示され、その直後に
「コード・コミットメッセージ等も含めて日本語で行ってください。当然 CLAUDE.md
ファイルも日本語です」と範囲が拡張された。第一言語が日本語であるため、思考と
レビューの速度を最大化する目的。

**How to apply**：
- 新規ファイル作成時、ユーザー向けテキストやコメントは日本語で書く
- 既存ファイルの編集時、追加・修正するコメント／ドキュメント部分は日本語で書く
  （既存の英語テキストの一律翻訳は不要）
- 識別子は英語の慣習を維持。日本語識別子は使わない
- ユーザーが個別の場面で別言語を求めた場合のみ、その範囲で切り替える

### 3.2 VS Code 統合機能を優先する

**ルール本文**：ツール選定で複数の選択肢がある場合、特に外部 CLI のインストールが
必要なものと VS Code 拡張で代替可能なものが並ぶとき、**VS Code 統合機能を第一
候補として提案する**。

具体例：
- GitHub 連携 → `gh` CLI ではなく `GitHub Pull Requests and Issues` 拡張＋
  Source Control の "Publish to GitHub"
- Git 操作 → ターミナルの `git` コマンド連打ではなく、VS Code の Source Control ビュー
- DB クライアント、API クライアント、Markdown プレビュー等 → 可能な限り
  VS Code 拡張を優先

**Why**：2026-05-01 にユーザーから「VS Code 上で Claude Code を実行しているので
GitHub に関してはこの環境で利便性がある方法がよい」と明示の指示があった。
VS Code 内で完結する方が認証情報・設定の共有がしやすく、普段のワークフローを
崩さない。

**How to apply**：
- 新ツール導入が必要な場面では、まず VS Code 拡張で実現できないか検討する
- CLI が必要な場合は理由（自動化、Claude 側からの代行操作など）を明示する
- 「Claude が代行」と「ユーザーが UI で実行」の住み分けを意識する。ユーザーの
  2〜3 クリックで済むことは UI 操作に任せる方が結果的に速いことがある
- ユーザーが「コマンドラインで進めたい」「自動化したい」と明示した場合はそちらを優先

### 3.3 既存資源優先・ゼロコスト重視

**ルール本文**：新規の有料サービスを提案する前に、ユーザーが既に保有・契約して
いるサービスおよび無料枠（レンタルサーバ、GitHub 無料アカウント、クラウド
ストレージ無料枠など）で要件を満たせるかを先に検討する。

**Why**：2026-05-02 に「目標達成を無料で行うことは可能か？」と問われ、この
対話を契機にサーバ型アーキテクチャからブラウザ完結型へ全面方針転換した経緯
がある。ゼロコスト達成が重要な制約として確立されている。

**How to apply**：
- インフラ／外部サービス／SaaS を提案する前に、既存資源での代替可否を確認する
- 具体的に何を保有しているかは **新プロジェクト開始時にユーザーへ確認する**
  （契約状況は時期によって変わる前提）
- やむを得ず有料サービスが必要なら、次を明示してから提案する
  - なぜ既存資源では不足するか
  - 必要な最小プランと月額の見積もり
  - 代替案（Self-host / OSS / 別アーキテクチャ）の有無

### 3.4 コミットを適宜行う

**ルール本文**：ユーザーから明示の許可がなくても、機能・論理単位の区切りで
`git commit` を実施する。`origin` が GitHub に向いている場合は push まで含めて
「コミット」として扱う。コミット粒度は「ひとまとまりの意味のある変更」とし、
1 ファイル 1 コミットなど過度な細分化はしない。

**Why**：2026-05-05 に「GitHub へのコミットは適宜行うようにしてくれ」と明示の
指示があった。Claude Code の既定動作（明示の指示がない限りコミットしない）
よりも、こまめなコミット＆push を優先する。履歴の粒度を保ち、ローカルの変更が
積み上がってロールバックしづらくなることを防ぐ目的。

**How to apply**：
- 機能・論理単位（初期化、UI 実装、動作確認、デプロイ準備など）ごとにコミットを切る
- `origin` が設定されており GitHub の場合は push まで実施する
- `--no-verify` でフックを skip しない、`--amend` で過去コミットを上書きしない
  といった基本ルールは厳守する
- ステージングは `git add <files>` で明示する。`git add -A` や `git add .` で
  意図しないファイル（`.env` など）を入れない
- コミットメッセージは日本語（章 3.1 に従う）。タイトルは簡潔に、本文で「なぜ」
  を補足する

## 4. CLAUDE.md テンプレ（プレースホルダ付き）

新プロジェクトの直下に置く `CLAUDE.md` の骨格。`{{...}}` の箇所をプロジェクト
固有情報で埋める。**章 3.1 の「使用言語」節は丸ごとそのまま貼り付け** することを
推奨（毎回書き直すと表現がブレるため）。

```markdown
# CLAUDE.md

このファイルは、本リポジトリで作業する Claude Code（claude.ai/code）への
指針を提供します。

## リポジトリの状態
{{repository_state}}
<!-- 例：仕様策定段階／実装中／運用中。実装の有無、現時点で「正」となる
     ファイル（仕様書、サンプル素材など）を 1〜3 文で書く。 -->

## プロダクト概要
{{product_summary}}
<!-- 1 段落でプロダクトの目的と主要な動き方。 -->

### 厳守すべき制約
{{constraints}}
<!-- 箇条書き。サイズ上限、対応形式、UI 要件、保持ポリシーなど。 -->

### 完了条件
{{done_criteria}}
<!-- 番号付き箇条書き。手動テスト、自動テスト、本番動作確認など。 -->

## 使用言語
（章 3.1 の「ルール本文」「例外」を本文としてそのままコピー）

## 確定済みの実装方針
{{tech_stack_and_architecture}}
<!-- 箇条書き。アーキテクチャ、フロント／バック／DB の技術選定、デプロイ先、
     特殊なバージョン pin、検証戦略など。 -->
```

### プレースホルダ一覧

| プレースホルダ | 何を書くか | 例（monorize 由来） |
|---|---|---|
| `{{repository_state}}` | 実装の有無、現時点で正となるファイル | 仕様策定段階／`project.md` と `sample/` のみが正 |
| `{{product_summary}}` | プロダクトの目的と主要な動き方 | 動画を **モノクロアニメ GIF** に変換するブラウザ完結型 SPA |
| `{{constraints}}` | サイズ上限・対応形式・UI 要件・保持ポリシー等 | 1 操作 1 ファイル／500 MB 推奨／GIF はモノクロ |
| `{{done_criteria}}` | テスト戦略・本番動作確認の基準 | 手動テスト＋10,000 件自動化＋本番／ステージング動作確認 |
| `{{tech_stack_and_architecture}}` | アーキテクチャ・技術選定・バージョン pin | Vite + React + ffmpeg.wasm 0.12.x／XServer 本番／GitHub Pages ステージング |

```markdown
履歴の粒度を保ち、ローカルの変更が積み上がってロールバックしづらくなることを
防ぐ目的。

**How to apply:**
- 機能・論理単位（プロジェクト初期化、UI 実装、動作確認、デプロイ準備など）
  ごとにコミットを切る
- リポジトリに `origin` が設定されており、GitHub の場合は push まで実施
- `--no-verify` でフックを skip しない、`--amend` で過去コミットを上書きしない
  などの基本ルールは引き続き厳守
- ステージングは `git add <files>` で明示。`git add -A` や `git add .` で
  意図せぬファイル（.env など）を入れない
- コミットメッセージは日本語（プロジェクトの言語方針に従う）。タイトルは
  簡潔に、本文で「なぜ」を補足する
```

### 5.2 抽象化版 1 件（既存資源優先）

monorize の memory には XServer／GitHub／Google Drive と具体名が入っているが、
別プロジェクトでは保有サービスが異なる可能性があるため、テンプレでは
プレースホルダ化する。**新プロジェクト開始時にユーザーへ「現在どのサービスを
保有・利用中か」を確認** してから埋める。

#### `feedback_prefer_existing_resources.md`

```markdown
---
name: {{project_name}} は既存資源優先・ゼロコスト重視
description: {{project_name}} プロジェクトでは新規有料サービスを提案する前に、ユーザー保有の既存資源で要件を満たせるか先に検討する
type: feedback
---
{{project_name}} プロジェクトでは、新規の有料サービスを提案する前に、
ユーザーがすでに保有している既存資源（{{existing_services}}）で要件を
満たせるかを先に検討する。

**Why:** ユーザーは過去プロジェクトで「目標達成を無料で行うことは可能か？」
と問い、その対話を契機に有料サービス前提のアーキテクチャから無料資源完結型へ
全面方針転換した経緯がある（2026-05-02）。ゼロコスト達成が **重要な制約** として
確立されている。

**How to apply:** インフラ・外部サービス・SaaS を提案するときは、まず既存資源
での代替可否を確認する。やむを得ず有料サービスが必要なら、
- なぜ既存資源では不足するか
- 必要な最小プランと月額の見積もり
- 代替案（Self-host / OSS / 別アーキテクチャ）の有無

を明示してから提案する。
```

`{{existing_services}}` の埋め方の例：
- monorize 案件の場合：`XServer レンタルサーバ、GitHub 無料アカウント、Google Drive 無料枠`
- 新プロジェクトの場合：開始時にユーザーへ確認した内容（例：`Vercel 無料枠、Cloudflare 無料枠、GitHub 無料アカウント`）

### 5.3 `MEMORY.md` のインデックス記法

`MEMORY.md` は本文ではなく **索引専用** のファイル。常にコンテキストに読み込ま
れるため、長くしすぎない（200 行以降は切り捨てられる）。

- 1 行 150 字以内
- 形式：`- [タイトル](file.md) — 一文の hook`
- 索引は時系列ではなく **トピック別** に並べる

例：

```markdown
- [すべての成果物を日本語で作成](feedback_language_japanese.md) — {{project_name}} ではユーザー応答・コード内コメント・コミット・ドキュメントを含むすべての生成テキストを日本語で書く（識別子と言語キーワードのみ英語）。
- [VS Code 統合機能を優先する](feedback_prefer_vscode_integration.md) — CLI と VS Code 拡張のどちらでも実現できる場合、VS Code 統合（Source Control、GitHub PR 拡張など）を第一候補として提案する。
- [既存資源優先・ゼロコスト重視](feedback_prefer_existing_resources.md) — {{project_name}} では新規有料サービス提案前に、保有済みの既存資源で要件を満たせるか先に検討する。
- [コミットを適宜行う](feedback_commit_proactively.md) — {{project_name}} では機能/論理単位の区切りでこまめに git commit & push を実施する。
```

## 6. `.claude/settings.local.json` の運用方針

### 基本方針
- 権限は **プロジェクトごとに必要に応じて追記** する（汎用テンプレ化はしない）
- 最初は空のまま始め、Claude Code が許可を求めてきたタイミングで追加する
- 個別の許可項目を Claude 側から自動追記する場合は、**そのプロジェクトに固有の
  コマンド** のみに留める（他プロジェクトに漏れる広すぎる許可を入れない）

### よく許可する系統（monorize で実際に使われた代表例）

新プロジェクトでも序盤に求められやすい系統。最初から `permissions.allow` に
入れておきたい場合の参考リスト。

| 系統 | 代表的な許可例 |
|---|---|
| git 基本操作 | `Bash(git status *)`, `Bash(git add *)`, `Bash(git commit *)`, `Bash(git push *)`, `Bash(git tag *)`, `Bash(git revert *)`, `Bash(git rm *)`, `Bash(git config *)`, `Bash(git check-ignore *)` |
| GitHub CLI | `PowerShell(gh *)`, `Bash(gh api *)`, `Bash(gh repo *)`, `Bash(gh run *)` |
| npm／Node | `PowerShell(npm *)`, `Bash(npm run *)`, `Bash(npm view *)`, `Bash(npx tsc *)`, `Bash(npx vite *)` |
| Web 系 | `WebFetch(domain:github.com)`, `WebFetch(domain:api.github.com)`, `WebSearch` |
| ファイル一覧／削除 | `PowerShell(Get-ChildItem *)`, `Bash(rm *)` |

### 入れない方が良いもの
- `Bash(*)` のような **無制限ワイルドカード**（誤操作の温床になる）
- `permissions.deny` ／ hooks ／ model 設定（本プロジェクト群では未使用のため、
  必要が発生した時点で再検討）

## 7. 本ガイド更新時のルール

### 追加の基準
新しい方針を本ガイドへ追加するのは、次のいずれかを満たすとき：
- ユーザーから **明示の恒久指示** があり、かつ
- **複数プロジェクトで再利用される見込み** がある（単発のタスク対応はガイドに
  入れない）

### 更新・撤回の手順
- 古くなった方針は、出元（指示があった日付や経緯）を本文に残したまま **更新** する
- 完全に撤回されたものは Git の履歴に残すかたちで **削除** してよい
- 本ガイドの編集自体も、章 3.1 と章 3.4 に従う（日本語で書く、こまめにコミットする）

### 章番号と memory ファイルの対応
本ガイドの章 3／章 5 と、実際の memory ファイル（`feedback_*.md`）の本文は
**同期させる**。片方だけ更新するとガイドと実環境が乖離するため、両方を同じ
コミットで揃える。
